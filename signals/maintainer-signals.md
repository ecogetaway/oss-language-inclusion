# Maintainer Signals

This file summarizes **patterns** from maintainer-side input and public maintainer statements linked in this repository. It is a processed view; raw input belongs in `maintainer-feedback/`.

## Observed Patterns

- Maintainers often frame i18n decisions as **capacity and process constraints**, not rejection of non-English users.
- Review bottlenecks are frequently tied to **native-language review depth** and release coordination across UI, errors, and docs.
- Teams commonly request **architecture readiness first** before accepting sustained locale expansion.
- Projects use practical triage (consolidation, duplicate closure, waiting states) when no stable i18n workflow exists.

## Scope Note

These are repository-level signals, not ecosystem-wide proof. See upstream-linked case studies in `case-studies/` for source context.

- Solo developer (Unity-based project): Absence of predefined i18n structure forces custom workarounds, increasing workload and long-term fragility
